## 2. 当前项目已完成的能力基线

根据 `docs/PROJECT_ARCHITECTURE.md`，当前已经具备以下基础能力：

### 已有核心功能
- 单基金查询与总览页
- 自选列表（本地 JSON 持久化）
- 持有列表（本地 JSON 持久化）
- 基金净值/估值/历史净值抓取
- 区间图表与技术指标展示
- 成本线叠加
- 持有录入、编辑、删除

### 已有技术边界
- 前端：React 18 + TypeScript + Vite + React Router + ECharts
- 后端：Node.js + Express + TypeScript
- 数据：以本地 JSON 持久化为主，不引入数据库
- 运行形态：本地个人工作台，不是公有 SaaS

### 这意味着什么
后续 PRD 的目标不是“从零做基金 App”，而是围绕当前已经成型的三层结构继续升级：

1. **增强单基金分析深度**
2. **增强列表管理、比较、筛选能力**
3. **增强组合诊断、导入、提醒、复盘能力**

## 3. 竞品调研结论摘要

本轮主要参考了这些公开产品/能力：

### 3.1 养基宝
公开信息显示，养基宝的价值点集中在：
- 多平台基金持仓汇总
- 图片识别/OCR 同步持仓
- 一站式管理持仓
- 盈亏分析
- 周报/月报
- 多账户账本

**对本项目的启发：**
- 当前 `financial` 已有“持有”能力，但还缺少“账户维度”“导入维度”“复盘维度”。
- 后续应补：多账户账本、导入中心、周/月报。

### 3.2 天天基金
公开资料显示，天天基金的成熟能力包括：
- 自选与持仓联动
- 基金详情页更多解释型分析
- 净值估算展示
- 条件选基、基金对比、组合工具

**对本项目的启发：**
- 当前 `financial` 的查询和总览已经有基础，但基金详情页仍偏“数据卡 + 图表”，不够“解释型”。
- 后续应补：基金详情增强、对比中心、条件选基。

### 3.3 蚂蚁理财金选 / 蚂蚁财富思路
公开资料显示，其关键方向在：
- 统一分析指标
- 多赛道、多维度选基
- 透明解释为什么入选
- 更强调收益、稳定性、性价比、投资管理等多维评估

**对本项目的启发：**
- 筛选不是只按收益排序，而应是多维评分和解释。
- 后续筛选功能需要“透明评分卡”，而不是黑盒推荐。

### 3.4 且慢全平台基金资产分析
公开资料显示，其强点在：
- 全平台基金资产汇总
- 穿透到底层股/债/商/现的大类资产
- 组合风险诊断
- AI 给出改进建议

**对本项目的启发：**
- 当前 `financial` 只有单基金和个人持有记录，没有“组合视角”。
- 后续应补：组合诊断、资产穿透、配置建议。

### 3.5 好买基金对比 / 同类工具
公开资料显示，成熟对比工具通常支持：
- 收益区间对比
- 风险指标对比
- 资产配置对比
- 基金经理/基金公司对比
- 重仓行业与重仓股对比

**对本项目的启发：**
- 对比功能必须做成独立中心，而不是塞到详情页里凑合。

## 4. 为什么是这 8 个 PRD

这 8 个功能不是随便列的，而是按当前项目的演化路径拆出来的：

1. **基金详情增强**：把现有核心页先做厚。
2. **高级自选与持有管理**：把日常使用频率最高的列表页做强。
3. **基金对比中心**：让用户能横向判断，不只看单只基金。
4. **条件选基与排行榜**：从“查已知基金”升级到“发现候选基金”。
5. **组合诊断与持仓穿透**：从“单只视角”升级到“组合视角”。
6. **多账户账本与导入中心**：解决真实使用中的录入成本与多平台资产割裂问题。
7. **提醒中心与自动化**：把工具从“查询型”升级为“持续服务型”。
8. **周报月报与复盘中心**：形成完整的投后复盘闭环。

## 5. 推荐开发顺序（严格执行）

## Phase 1：夯实现有主链路
### 1）`01-基金详情增强_PRD.md`
优先级：P0

原因：
- 当前总览页已经是核心入口，增强它的投入产出比最高。
- 不依赖新的全局复杂数据模型。
- 开发完成后，立刻提升“查一只基金”的决策价值。

### 2）`02-高级自选与持有管理_PRD.md`
优先级：P0

原因：
- 自选 / 持有是当前项目第二高频模块。
- 可直接复用已有 JSON 持久化能力。
- 能为后续的诊断、提醒、复盘提供更干净的数据基础。

## Phase 2：从查看走向比较与发现
### 3）`03-基金对比中心_PRD.md`
优先级：P1

原因：
- 用户从“我看一只”走向“我比几只”。
- 需要复用详情查询和计算逻辑，但不强依赖选基系统。

### 4）`04-条件选基与排行榜_PRD.md`
优先级：P1

原因：
- 对比功能解决“在候选中选”，筛选功能解决“从市场里找”。
- 会引入新的市场基金池缓存与刷新逻辑，因此放在对比之后更稳。

## Phase 3：从单只基金走向组合管理
### 5）`05-组合诊断与持仓穿透_PRD.md`
优先级：P2

原因：
- 需要依赖“持有数据更完整”“基金信息更丰富”。
- 这是从工具型产品迈向专业工作台的关键分水岭。

### 6）`06-多账户账本与导入中心_PRD.md`
优先级：P2

原因：
- 解决真实使用中的迁移和录入问题。
- 完成后，组合诊断和复盘会有更完整数据。

## Phase 4：形成持续服务闭环
### 7）`07-提醒中心与自动化_PRD.md`
优先级：P3

原因：
- 需要稳定的数据刷新和更清晰的规则模型。
- 如果太早做，容易变成提醒垃圾场。

### 8）`08-周报月报与复盘中心_PRD.md`
优先级：P3

原因：
- 依赖前面多个模块的数据沉淀。
- 最适合放在后面整合输出价值。

## 7. 开发约束（很重要）

后续 agent 开发时，请遵守以下原则：

### 7.1 不要破坏当前本地优先架构
- 默认仍使用本地 JSON 持久化。
- 不要在没有明确必要时引入数据库。
- 不要强行上复杂状态管理库。

### 7.2 先做“能闭环”的功能，再做“看起来高级”的功能
- 先把查询、列表、对比、筛选闭环做扎实。
- 再做组合诊断、导入、提醒、复盘。

### 7.3 评分与建议必须可解释
- 不允许直接输出“推荐买入”这类黑盒结论。
- 需要告诉用户：评分来自哪些维度，强项弱项在哪里。

### 7.4 默认先做本地版、单用户版
- 先不要考虑登录、多用户、云同步、权限系统。
- 如果未来扩展，再单开 PRD。

### 7.5 优先复用现有模块
- 现有 `FundSummaryCard`、`ChartPanel`、`src/utils/fund.ts`、`server/fund-service.ts` 都应优先扩展，而不是推倒重做。

## 9. Definition of Done（每个阶段都要满足）

每个 PRD 落地时，至少满足：

1. 前端页面或模块真实可用，不是静态壳子。
2. API 和数据结构完整闭环。
3. 错误态、空状态、加载态有处理。
4. 本地 JSON 文件模型经过兼容性处理，不破坏旧数据。
5. 文档同步更新：
   - `docs/PROJECT_ARCHITECTURE.md`
   - 本阶段对应的 PRD 状态说明
6. 若功能引入新指标或评分，页面中必须有解释文案。
