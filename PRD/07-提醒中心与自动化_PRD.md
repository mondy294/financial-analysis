# PRD 07：提醒中心与自动化

> 优先级：P3
> 
> 目标：把 `financial` 从“手动打开才有价值”升级成“能持续提醒和陪伴”的工具。

---

## 1. 背景

当前项目更像一个查询台：
- 用户主动打开
- 手动查基金
- 手动看持有

但真实使用中，长期价值更强的是：
- 回撤提醒
- 成本线附近提醒
- 估值异动提醒
- 自选池重点关注提醒
- 周期性复盘提醒

竞品启发：
- 养基宝用户评论多次提到“实时估值、智能提醒”
- 头部财富平台普遍有投后陪伴机制

---

## 2. 目标

建设一个本地版提醒中心，支持：
1. 规则提醒配置
2. 提醒消息中心
3. 浏览器通知（可选）
4. 周期性摘要提醒
5. 与自选/持有/诊断/复盘联动

---

## 3. 非目标

本阶段不做：
- 短信/微信/邮件推送
- 云端推送
- 高频实时毫秒级通知

当前形态是本地桌面工作台，优先做：
- 站内消息
- 浏览器通知
- 页面红点
- 手动刷新下的规则触发

---

## 4. 提醒类型设计

## 4.1 基金价格/估值类提醒
- 当日估值涨跌幅超过阈值
- 最新净值高于/低于用户自定义阈值
- 接近或跌破成本净值

## 4.2 风险类提醒
- 近 X 日回撤超过阈值
- 波动率进入高风险区间
- 组合科技/行业暴露超过阈值（依赖组合诊断）

## 4.3 行为类提醒
- 自选基金连续 N 天未查看
- 持有记录长时间未更新
- 有缺失字段影响诊断精度

## 4.4 周期类提醒
- 每周复盘提醒
- 每月月报提醒
- 每周刷新基金池提醒

---

## 5. 用户场景

### 场景 A：我不想天天盯盘
用户希望：
- 只有在重要波动时提醒我
- 不想被无意义消息轰炸

### 场景 B：我想围绕成本线管理仓位
用户希望：
- 当基金净值接近成本线时提醒
- 当浮盈/浮亏达到某阈值时提醒

### 场景 C：我需要定期复盘
用户希望：
- 每周收到一次摘要，知道哪些基金值得重点看

---

## 6. 核心功能拆解

## 6.1 提醒规则管理
### 字段建议
- `id`
- `scopeType`：fund / watchlist / holdings / portfolio / system
- `scopeValue`：基金代码或组合标识
- `ruleType`
- `operator`
- `threshold`
- `enabled`
- `cooldownHours`
- `createdAt`
- `updatedAt`

### 示例
- 基金 270042 当日估值涨跌幅 > 2.5%
- 基金 161725 最新净值 < 成本净值
- 组合科技暴露 > 40%
- 每周日 20:00 提醒我做周报

---

## 6.2 消息中心
### 页面入口
- 顶部铃铛图标
- 独立页面：`/notifications`

### 消息结构
- 标题
- 内容摘要
- 关联对象
- 触发时间
- 已读 / 未读
- 跳转链接

### 示例
- “招商中证白酒估值下跌 2.8%，已触发你的阈值提醒。”
- “你的组合科技主题暴露升至 47%，高于预设阈值。”

---

## 6.3 浏览器通知（可选）
### V1 要求
- 用户手动授权后可开启
- 默认关闭
- 仅对高优先级提醒弹出

### 设计原则
- 不做骚扰式通知
- 同一规则在冷却时间内不得重复打扰

---

## 6.4 提醒中心首页摘要
### 展示内容
- 今日新提醒数
- 高优先级提醒数
- 近 7 日触发最多的规则
- 当前已启用规则数

---

## 6.5 周期性摘要提醒
### 类型
- 每周日生成“本周待复盘基金清单”
- 每月初生成“上月持仓摘要待查看”

### 输出方式
- 先写入消息中心
- 若用户开启浏览器通知，再弹提示

---

## 7. 页面与交互建议

## 7.1 规则创建入口
从多个地方可以创建提醒：
- 基金详情页：为当前基金创建提醒
- 持有页：为持仓项创建提醒
- 组合诊断页：为组合风险项创建提醒
- 提醒中心：统一管理所有规则

## 7.2 规则模板
为降低配置成本，提供快捷模板：
- 涨跌幅提醒
- 成本线提醒
- 回撤提醒
- 周报提醒

---

## 8. 数据模型设计

### alerts.json（新增）
```json
{
  "items": [
    {
      "id": "alert_001",
      "scopeType": "fund",
      "scopeValue": "270042",
      "ruleType": "estimate_change_pct",
      "operator": ">=",
      "threshold": 2.5,
      "enabled": true,
      "cooldownHours": 12,
      "createdAt": "2026-04-15T13:00:00.000Z",
      "updatedAt": "2026-04-15T13:00:00.000Z"
    }
  ]
}
```

### notifications.json（新增）
```json
{
  "items": [
    {
      "id": "notice_001",
      "level": "high",
      "title": "招商中证白酒估值下跌 2.8%",
      "content": "已触发你设置的估值波动提醒",
      "scopeType": "fund",
      "scopeValue": "161725",
      "read": false,
      "createdAt": "2026-04-15T13:20:00.000Z"
    }
  ]
}
```

## 9. API 设计

- `GET /api/alerts`
- `POST /api/alerts`
- `PATCH /api/alerts/:id`
- `DELETE /api/alerts/:id`
- `POST /api/alerts/check` —— 手动触发一次规则检查
- `GET /api/notifications`
- `PATCH /api/notifications/:id/read`
- `POST /api/notifications/read-all`

---

## 10. 规则执行策略

### V1
- 在页面加载时检查一次
- 用户点击刷新时检查一次
- 用户进入提醒中心时检查一次

### V2
- 本地定时轮询
- 与周报/月报定时任务联动

### 冷却机制
同一规则在 `cooldownHours` 内只生成 1 条高优先级消息。

---

## 11. 验收标准

### 功能验收
- 可创建多种类型的提醒规则
- 可在消息中心查看触发记录
- 可标记已读 / 全部已读
- 可从基金详情 / 持有页快速创建提醒

### 体验验收
- 消息数量受控，不刷屏
- 规则创建过程简单
- 提醒内容能明确说明为什么触发

### 技术验收
- 冷却机制有效
- 数据持久化稳定
- 不阻塞原有页面查询流程

---

## 12. 建议实施拆分

先做 alerts.json 与规则 CRUD。

### Step 2
做消息中心和站内提醒。

### Step 3
做规则检查逻辑与冷却机制。

---

## 13. 完成后的价值

完成后，`financial` 会从“我想起来才打开”变成“有值得注意的事情时会叫我一声”。

这会明显增强长期使用价值。
