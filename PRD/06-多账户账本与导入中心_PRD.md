# PRD 06：多账户账本与导入中心

> 优先级：P2
> 
> 目标：解决真实使用中的“持仓录入成本高、多个平台资产割裂、迁移麻烦”问题。

---

## 1. 背景

当前项目的持有记录依赖手动填写：
- code
- 状态
- 收益率
- 持仓金额
- 成本净值
- 备注

这对轻度使用还行，但一旦真实持仓分散在多个平台：
- 支付宝
- 理财通
- 天天基金
- 券商账户
- 家庭多个账户

手动维护会迅速变得又烦又脆。

竞品启发：
- 养基宝：多账户账本 + 图片识别/OCR 同步持仓 + 多平台汇总
- 且慢：通过基金 E 账户导出，再做统一分析

---

## 2. 目标

建设一个“多账户账本与导入中心”，支持：
1. 多账户管理
2. 批量导入持仓
3. CSV/表格导入
4. 截图/OCR 导入预留
5. 导入校验与确认
6. 按账户查看持仓和汇总

---

## 3. 分阶段目标

## V1（本 PRD 交付范围）
- 多账户定义
- 手动批量导入
- CSV 导入
- 导入预览 / 字段映射 / 冲突处理

## V2（预留）
- 图片 OCR 导入
- 第三方导出模板识别
- 更复杂的平台模板适配

这样能避免一上来就掉进 OCR 细节泥潭里。

---

## 4. 用户场景

### 场景 A：我在多个平台都买了基金
用户希望：
- 区分不同平台和账户
- 汇总整体持仓
- 还能看每个平台单独表现

### 场景 B：我已有一份持仓表
用户希望：
- 直接导入 CSV
- 不想一条条手敲

### 场景 C：我想把现有项目迁移给家人或另一个设备
用户希望：
- 导入 / 导出都标准化

---

## 5. 功能拆解

## 5.1 多账户模型
### 账户字段
- `id`
- `name`：如“支付宝主账户”“理财通账户”“家庭账户”
- `platform`：支付宝 / 理财通 / 天天基金 / 券商 / 手工账本 / 其他
- `currency`：默认 CNY
- `note`
- `createdAt`
- `updatedAt`

### 使用方式
- 每条持有记录必须可选绑定一个账户
- 不绑定时归为“默认账户”

---

## 5.2 导入中心页面
### 路由
新增：`/import-center`

### 页面结构
1. 导入方式选择
2. 模板说明
3. 文件上传区
4. 字段映射区
5. 预览表格
6. 冲突处理区
7. 提交导入结果页

---

## 5.3 导入方式
### V1 支持
- 手动批量粘贴（文本/表格）
- CSV 上传
- JSON 上传（供高级用户与备份恢复）

### V2 预留
- 图片 OCR 上传
- 平台截图识别
- 平台导出单据模板识别

---

## 5.4 标准导入字段
### 最小必填字段
- `code`

### 推荐字段
- `accountName`
- `status`
- `holdingReturnRate`
- `positionAmount`
- `costNav`
- `tags`
- `note`

### 模板示例
```csv
code,accountName,status,holdingReturnRate,positionAmount,costNav,tags,note
270042,支付宝主账户,持有中,4.21,10297.2,7.0088,"底仓|定投",长期观察仓
161725,理财通账户,观察仓,-1.32,5000,1.5230,"医疗|观察",等待回撤
```

---

## 5.5 字段映射与预览
### 目标
不同用户的导入表头可能不一致，因此不能要求死模板。

### 功能要求
- 上传后自动识别列头
- 用户可手动映射到标准字段
- 提交前必须展示预览
- 对异常项给出警告

### 异常示例
- 基金代码不是 6 位
- 成本净值不是数字
- 持仓金额为空
- 状态不在允许枚举中

---

## 5.6 冲突处理策略
### 典型冲突
同一个账户下同一基金已存在旧记录。

### 用户可选策略
- 覆盖旧记录
- 跳过已有记录
- 保留两条并提示后续手工合并（不推荐默认）

### 默认策略
- 同账户 + 同 code → 视为同一持仓，默认“更新覆盖”

---

## 5.7 按账户视图
### 持有页增强
持有页新增：
- 账户筛选器
- 按账户分组视图
- 每个账户的小计摘要

### 摘要字段
- 账户总持仓金额
- 账户加权收益率
- 账户基金数量

---

## 5.8 导出能力
### 最低要求
支持导出：
- 当前 holdings 为 CSV
- 当前 holdings 为 JSON

### 价值
- 方便备份
- 方便换机
- 方便后续导入其他分析工具

## 6. 数据模型设计

```json
{
  "items": [
    {
      "id": "acc_alipay_main",
      "name": "支付宝主账户",
      "platform": "支付宝",
      "currency": "CNY",
      "note": "个人主账户",
      "createdAt": "2026-04-15T12:00:00.000Z",
      "updatedAt": "2026-04-15T12:00:00.000Z"
    }
  ]
}
```

## 6.2 holdings.json 扩展
```json
{
  "items": [
    {
      "code": "270042",
      "accountId": "acc_alipay_main",
      "status": "持有中",
      "holdingReturnRate": 4.21,
      "positionAmount": 10297.2,
      "costNav": 7.0088,
      "tags": ["底仓", "定投"],
      "note": "长期观察仓",
      "updatedAt": "2026-04-15T04:12:57.647Z"
    }
  ]
}
```

## 6.3 import-jobs.json（新增）
记录每次导入任务的结果：
- 来源文件名
- 时间
- 成功条数
- 失败条数
- 冲突条数
- 错误详情

## 9. OCR 预留设计（不要现在实现）

### 为什么要预留
竞品里这块很有吸引力，但也是坑最多的部分。

### 当前建议
在导入中心中先放一个“即将支持 OCR 导入”的占位模块，并预留接口：
- `POST /api/import/ocr-preview`

但本阶段不落地真实 OCR。

这样既给后续 agent 留出扩展位，也不把当前阶段拖死。

## 10. 验收标准

### 功能验收
- 能创建多个账户
- 能导入 CSV 并映射字段
- 能预览导入结果并处理冲突
- 导入后可在持有页按账户查看
- 能导出当前持有数据

### 兼容性验收
- 旧 holdings 数据无 accountId 时仍可读
- 系统能自动归入默认账户

### 体验验收
- 导入流程不超过 4 步核心操作
- 错误提示清晰，不让用户猜错在哪
